# Plan: 添加 QQ / 微信 / 飞书聊天记录查询

## 背景分析

### 现有架构
- `backend/tools/ai_session_manager/` 是核心工具，通过 `config.py` 配置数据源路径（kilo/opencode），通过 `database.py` 访问 SQLite 数据库，通过 `routes/` 提供 REST API
- 前端在 `frontend/src/pages/tools/AiSessionManager.tsx` 中通过 `DataSource` 类型切换 kilo/opencode 数据源
- 工具自动发现机制：`backend/main.py` 扫描 `tools/` 目录，每个子目录包含 `meta.json` + `router.py` 即自动注册

### 数据存储现状（本机实测）
| 平台 | 路径 | 格式 | 加密 |
|------|------|------|------|
| **微信** | `C:\Users\zccyman\Documents\WeChat Files\{wxid}\Msg\` | SQLite (SQLCipher) | 全加密（header 也被加密） |
| **QQ** | `C:\Users\zccyman\Documents\Tencent Files\1051775020\nt_qq\nt_db\` | SQLite (加密) | 内容加密 |
| **飞书** | `C:\Users\zccyman\AppData\Roaming\LarkShell\` | 专有格式 | 不可访问 |

关键数据库文件：
- 微信: `MicroMsg.db`（联系人）, `ChatMsg.db`（单聊消息）, `Multi/MSG0-10.db`（分片消息）
- QQ: `nt_msg.db`（消息）, `group_msg_fts.db`（群消息全文搜索）, `group_info.db`（群信息）
- 飞书: `persistent_storage.enc.db`（加密存储）

**结论：三个平台的本地数据都经过加密，不能像 kilo/opencode 那样直接读取 SQLite。**

## 方案设计

### 核心思路：一键备份 → 解密导入 → 统一查询

采用三阶段架构：

1. **一键备份**：将原始加密数据库文件备份到 `D:\backup\{platform}\` 目录，支持增量备份（只复制新增/变更文件）
2. **解密导入**：从备份目录读取加密数据库 → 解密 → 导入到 workbench.db 统一数据模型
3. **查询展示**：后端提供统一查询 API，前端展示聊天记录

### 增量备份策略

- **全量备份**：首次备份，复制所有数据库文件
- **增量备份**：对比源文件和备份文件的 `mtime + size`
  - 新增文件：直接复制
  - 修改文件：覆盖更新
  - 未变文件：跳过
  - 排除 `-shm`/`-wal` 临时文件（这些是 WAL 模式的临时文件）
- **元数据文件** `backup_meta.json` 记录每次备份的时间、文件列表、状态

### Task 2: 实现备份核心模块（`backup.py`）

这是整个功能的起点——一键备份按钮对应的后端逻辑。

#### 备份流程
```
用户点击"一键备份"
  → 前端 POST /api/tools/chat_records/backup/{platform}
  → 后端 backup.py 执行：
    1. 读取源路径（微信/QQ/飞书的本地数据目录）
    2. 确定备份目标路径（D:\backup\chat-records\{platform}\）
    3. 扫描源目录，收集所有 .db 文件
    4. 对比 backup_meta.json 中的记录：
       - 新增文件 → 加入复制列表
       - mtime 或 size 变化 → 加入复制列表
       - 未变化 → 跳过
    5. 执行文件复制（排除 -shm/-wal 文件）
    6. 更新 backup_meta.json
    7. 返回备份结果（新增数、更新数、跳过数）
```

#### 增量备份核心逻辑
```python
def incremental_backup(source_dir: str, backup_dir: str, meta: dict) -> BackupResult:
    """
    增量备份：
    1. 扫描 source_dir 下所有 .db 文件（排除 -shm/-wal）
    2. 对每个文件，检查 meta 中记录的 mtime+size
    3. 仅复制新增或变更的文件
    """
    # meta 结构示例：
    # {
    #   "last_backup": "2026-04-10T12:00:00",
    #   "files": {
    #     "Msg/MicroMsg.db": {"mtime": 1234567890, "size": 1024000},
    #     "Multi/MSG0.db": {"mtime": 1234567890, "size": 2048000}
    #   }
    # }
```

#### 删除源数据功能
```python
def delete_source_data(platform: str, source_dir: str, backup_dir: str) -> DeleteResult:
    """
    备份完成后，用户可选择删除源数据以释放磁盘空间。
    
    安全措施：
    1. 验证备份目录存在且有有效数据（检查 backup_meta.json）
    2. 验证备份文件完整性（比对备份文件数量与 meta 记录是否一致）
    3. 只删除已成功备份的文件（对比备份前后的文件列表）
    4. 不删除目录本身，只删除已备份的 .db 文件
    5. 返回删除结果（删除文件数、释放空间大小）
    
    注意：
    - 微信/QQ 运行时无法删除锁定的文件，需要提示用户关闭应用
    - 删除前需二次确认（前端弹出确认对话框，输入平台名称确认）
    - 删除操作不可逆，必须有充分的安全校验
    """
```

**删除流程**：
```
用户点击"删除源数据"
  → 前端弹出确认对话框（要求输入平台名称如 "wechat" 确认）
  → POST /api/tools/chat_records/backup/{platform}/delete-source
  → 后端执行：
    1. 校验备份目录存在且 backup_meta.json 有效
    2. 校验备份文件完整（已备份文件数 > 0）
    3. 遍历备份 meta 中记录的源文件列表
    4. 逐个删除源 .db 文件（跳过被锁定的文件）
    5. 记录删除日志到 backup_meta.json
    6. 返回删除结果（成功数、失败数、释放空间）
```

### Task 6: 添加解密辅助脚本（Windows 侧运行）

- **使用方式**：
  ```bash
  python scripts/decrypt_wechat.py --extract-key
  python scripts/decrypt_wechat.py --decrypt-all --input "D:\backup\chat-records\wechat"
  ```

#### `scripts/decrypt_qq.py`
- 同理，从 `D:\backup\chat-records\qq\` 解密
- 输出到 `data/qq_decrypted/`

## 风险和注意事项

1. **法律风险**：微信曾对 PyWxDump 发律师函。解密脚本仅供个人数据管理使用，不应公开分发。
2. **pysqlcipher3 编译**：在 WSL 上编译 pysqlcipher3 可能需要安装 `libsqlcipher-dev`。也可使用预编译的 `sqlcipher3`。解密脚本在 Windows 上运行时可以使用 Windows 版 sqlcipher。
3. **数据库锁定**：微信/QQ运行时会锁定数据库文件。**备份机制正好解决这个问题**：复制主 `.db` 文件（排除 `-shm`/`-wal`），即使应用正在运行也能获取数据。
4. **飞书API限制**：只能获取 Bot 加入群组后的消息，历史消息无法获取。飞书功能作为可选模块。
5. **QQ数据库格式**：NTQQ 的数据库加密方式可能随版本变化，需要持续维护。
6. **密钥提取**：微信/QQ的密钥提取需要特定版本的偏移量，不同版本可能不同。脚本需要支持常见版本。
7. **备份安全性**：备份的是加密数据库原始文件，即使备份文件泄露也无法直接读取，安全性有保障。
8. **删除源数据风险**：删除不可逆，必须有充分的安全校验。微信/QQ运行时文件被锁定无法删除，需提示关闭应用。删除后不影响应用运行（应用会自动重建空数据库），但历史聊天记录在应用中将消失。

## 关键技术决策

1. **备份优先**：先备份原始加密文件到 `D:\backup\`，再从备份中解密。好处：
   - 不直接操作原始数据，安全
   - 解决应用运行时锁定文件的问题
   - 增量备份效率高
   - 备份文件就是加密的原始文件，安全性好
2. **解密 vs API**：微信和QQ优先使用本地数据库解密（数据完整），飞书使用API（唯一途径）
3. **导入 vs 直接读**：采用导入到 workbench.db 的方式，因为：
   - 避免每次查询都要解密
   - 可以建立 FTS 索引加速搜索
   - 数据模型统一，查询逻辑简单
   - 支持离线查询
4. **独立脚本 vs 集成**：解密脚本独立运行（不在 FastAPI 进程中），因为需要 Windows 环境 + 进程访问权限
5. **增量备份**：基于文件 mtime + size 对比，简单可靠，无需计算文件哈希
6. **删除源数据**：仅在备份成功后可用，要求二次确认（输入平台名称），有完整性校验。删除的是已备份的原始 .db 文件，释放磁盘空间
